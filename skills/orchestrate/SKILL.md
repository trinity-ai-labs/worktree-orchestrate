# Orchestrate — release-branch worktree workflow

Our main dev workflow: we work off an integration branch (for Trinity, `release/x.x.x` — discover the active one with `git branch --list 'release/*'` / the current branch; never hardcode a version). For each task we cut a git worktree, do the work there, commit, push, and open a PR back into the integration branch. After a PR merges we **sync the local integration branch first**, then clean up the merged branch and its worktree.

This is project-agnostic. The per-project specifics — which env files to symlink, the install command, the gate command, the docs fast-path, and the conventions to bake into briefs — live in a config the tooling reads (see *Per-project config* below), not in this playbook.

**Default to planning for parallelization.** Given a batch of work, the first move is to decompose it into independent tasks that can run concurrently in their own worktrees — don't default to doing everything sequentially yourself. Spin up as many worktrees/branches as the work needs.

## First: which role are you?

Before acting, decide whether you are the **orchestrator** or an **implementer** — they behave very differently.

- **ORCHESTRATOR** when the user asks you to *orchestrate / coordinate* work, to *complete or work a GitHub issue*, to *execute a plan*, or hands you a batch of tasks. You are the coordinator. **Do NOT write the implementation yourself.** You decompose the work, create + verify worktrees, dispatch an implementer sub-agent into each (in parallel when independent), then run the review loop and merge. You hold the plan, the reviews, and the merges — not the file edits.

- **IMPLEMENTER** when you are a dispatched sub-agent, or the user *directly tells you to implement / build / fix* a specific thing. You do the actual code in your assigned worktree, get it green, open a PR, and hand back. **You do not merge your own PR** — the orchestrator reviews and merges.

## Per-project config

The worktree tooling is generic; each project declares its specifics in a bash file at `~/.worktrees/config/<project>.sh`, where `<project>` is the repo's directory name. The helper sources it; the orchestrator reads it too (e.g. `cat ~/.worktrees/config/<project>.sh`) to learn the gate and the brief conventions. Keys:

- `ENV_FILES` — gitignored env files to symlink from the main checkout into each worktree.
- `INSTALL_CMD` — install command run inside a new worktree (worktrees don't share `node_modules`).
- `GATE_CMD` — the green gate an implementer runs before opening a PR (the orchestrator's backstop).
- `DOCS_BRANCH_PREFIX` / `DOCS_NOTE` — branches under this prefix skip env + install (a fast path for deps-free work like markdown).
- `BRIEF_CONVENTIONS` — project conventions to bake into every dispatched implementer brief (framework skills, compat policy, comment style, simplify-then-gate, etc.).

A repo with no config still works — the helper makes a bare worktree (no env, no install). To onboard a new project, drop a `~/.worktrees/config/<project>.sh` declaring the keys above.

## Worktree creation (both roles depend on this being done right)

**⛔ HARD BAN — the Agent tool's `isolation: "worktree"` parameter (and ANY harness / auto worktree provisioner) is FORBIDDEN. Never use it.** It (a) creates the worktree under `.claude/worktrees/agent-*` (violates invariant 1 below) and (b) repeatedly seeds it at a STALE base — far behind the integration tip (observed ~1000+ commits behind, a previous-minor-version commit) — which has burned us over and over (agents waste cycles re-basing, or stale code silently leaks in). There is exactly ONE way to make a worktree: the helper script below, then **verify HEAD yourself**, THEN dispatch a plain agent (no `isolation`) pointed at that worktree path.

**Helper script — `~/.worktrees/setup-worktree.sh`** (a personal tool, NOT repo-tracked — edit it directly, no worktree/PR needed; don't search the repo for it). It auto-detects the project and reads its config, so it works the same for any repo.

Usage: `setup-worktree.sh <branch> <base>` — both **required**. Run it from **anywhere inside the target repo** (the root, a subdirectory, a monorepo subpackage, or even a linked worktree — it walks up to the repo's common gitdir). If cwd isn't inside the repo, prefix `REPO=/path/to/repo`.
- `<branch>` = full name of the branch to create (any prefix: `feat/…`, `fix/…`, `refactor/…`, `docs/…`); the worktree dir is named after the segment past the last slash (`feat/toasts-top-right` → `toasts-top-right`).
- `<base>` = the branch to fork from, with **no default** (integration branches roll over often, so a hardcoded default just goes stale); it errors early with a `git fetch` hint if `<base>` isn't a local ref yet.

It creates the worktree at `~/.worktrees/<project>/<branch-leaf>`, symlinks the config's `ENV_FILES`, and runs `INSTALL_CMD` — automating invariants 1, 3, 4 below. A branch under `DOCS_BRANCH_PREFIX` takes the fast path (env + install skipped).

**The four invariants** (the helper covers 1, 3, 4 — step 2 is always on you):
1. The worktree lives under `~/.worktrees/<project>/`, **never** under `.claude/worktrees`.
2. **Verify its HEAD == the base tip BEFORE dispatching any agent** — run `git -C <worktree> rev-parse HEAD` and confirm it equals `git rev-parse origin/<base>` (e.g. `origin/release/x.x.x`). The helper does **not** verify this for you — sync the base first, then check. A mismatch means STOP and fix the base; do not dispatch.
3. Env files are symlinked into the worktree (they aren't carried over automatically).
4. Deps are installed (worktrees don't share `node_modules`, and the gate can't run without them).

## Orchestrator

### Dispatch
Decompose the batch into independent tasks. For each: create + **verify** a worktree (above), then **dispatch a plain implementer sub-agent** (no `isolation`) pointed at that worktree path — in parallel whenever the tasks are independent. Spin up as many worktrees/branches as the work needs — but mind the gate-lock bottleneck: if many slices will finish and gate at roughly the same time, see *On the gate → wide independent fan-outs* before fanning out, or they will serialize N full gates on one lock and start stranding PRs.

**Pick the model tier per task — match cost to difficulty.** Pass `model: "sonnet"` to the Agent tool for the *bulk* of implementer dispatches: well-scoped, mechanical, or moderate work (wiring, UI from a clear spec, a route/op that mirrors an existing one, tests, a refactor with a known shape). Reserve `model: "opus"` for genuinely HARD slices: subtle algorithms, ambiguous or design-heavy work, tricky concurrency/lifecycle, security-sensitive code, or large cross-cutting changes where a wrong approach is expensive to unwind. When unsure, **start Sonnet** — a Sonnet PR you have to redirect is cheaper than Opus on routine work; escalate to Opus only if the task proves harder than scoped. Trade-off: the easier the model, the **more explicit the brief** must be — exact files, patterns to copy, hard do-not-touch boundaries, research-first steps. Clarity substitutes for model strength, so a Sonnet brief should read as near-deterministic steps, not "figure it out."

**Open every brief with "Step 0: invoke the `orchestrate` skill (via the Skill tool) and act as the IMPLEMENTER."** The agent then reads the generic implementer playbook from the *Implementer* section itself — cheap-checks-during / heavy-gate-at-end, the zero-warnings "don't be lazy" bar, push-open-PR-report, never-rebase / never-self-merge, and stop-and-report. **Do NOT re-paste any of that into the brief** — it's already in the skill the agent just loaded, and duplicating it bloats the brief and drifts out of sync.

**This skill is written for you, the orchestrator — you are the one who has fully internalized it, and your job is to translate the parts implementers must execute correctly into their brief.** Step 0 ("invoke the `orchestrate` skill and act as the IMPLEMENTER") covers simple absolute prohibitions fine — never rebase, never self-merge — an agent either does or doesn't do those. It does NOT cover (a) a multi-step procedure that has one correct order and one wrong, silently-incomplete order with no error or red gate to flag the miss, or (b) a rule that overrides a harness-level default the agent would otherwise follow unprompted. Three items have exactly one of those two shapes, and you paste all three into every brief close to verbatim rather than relying on Step 0 to carry them: the **commit-last ordering** and the **gate-wait mechanism** (shape a), and **no AI attribution** (shape b). Treat any future addition with either shape the same way.

Your brief carries the **task-specific context the skill can't know**, plus the three items above — keep the rest lean:
- **Step 0 skills.** Invoke `orchestrate` (as implementer), plus any framework reference skill the task touches — name it as an explicit first step, not "use it." For Trinity — **app = Solid, sidecar = Effect**: Effect-TS on the sidecar (services, layers, HTTP routes, typed errors) → "invoke the `effect` skill"; SolidJS UI on the app (`solid-js`, `@solidjs/router`, `@tanstack/solid-query`, Kobalte) → "invoke the `solid` skill"; a full-stack slice → both. Sonnet agents won't reach for these on their own, so spell them out.
- **The task + the worktree.** What to build/fix, the assigned worktree's absolute path, and the concrete files/patterns to copy. The weaker the model, the more explicit — near-deterministic steps for Sonnet, not "figure it out."
- **Scope + hard do-not-touch boundaries.** The other slices'/phases' core files this task must NOT edit. This is the part only you hold from the decomposition; the skill cannot supply it.
- **Project conventions for this slice.** The relevant bits of the project's `BRIEF_CONVENTIONS` (`~/.worktrees/config/<project>.sh`) and `AGENTS.md` — compat policy, comment style. (Trinity: pre-launch forward-only, no backwards-compat shims; comments explain the mechanism.)
- **Gate mode for this wave.** Whether the full `GATE_CMD` is their wrap-up signal, or this is a capped / integration-gated fan-out where they push after only a scoped check (see *wide independent fan-outs* below). The skill defaults to the full gate; only you know if this wave is capped.
- **The commit-last ordering.** `/simplify` acts only on the *uncommitted* working-tree diff, so the order is load-bearing: code committed early, or committed all at once before simplifying, is invisible to the pass — it reports "nothing actionable" against a clean tree whether or not the code actually needed cleanup. Paste this into every brief:

  > Do NOT commit as you go. Write the ENTIRE change in your worktree first — uncommitted. Cheap checks while you work are fine (format, a scoped lint/typecheck, a single targeted test file) but don't commit yet. When the code is done, run `/simplify` over your full uncommitted diff and fold its fixes in. THEN run `GATE_CMD` once and get it green. ONLY THEN commit — in logical, self-contained blocks (a type + its plumbing, a behavior + its tests), never one giant commit and never as-you-go.

- **The gate-wait mechanism.** The gate lock has no fairness or ordering, so an unbounded wait can strand committed work with no PR. Paste this into every brief:

  > **The mechanism is simple: finish your code, then check the lock — if it's free, run the gate; if it's held, wait a bounded window and retry. Never wait unbounded — that strands committed work with no PR.** Concretely:
  >
  > 1. **Check the lock.** `LOCK="$(node -e 'process.stdout.write(require("os").tmpdir())')/trinity-test-gate.lock"; PID="$(cat "$LOCK/pid" 2>/dev/null)"; ps -p "$PID" -o pid=,ppid=,command= 2>/dev/null || echo "lock free / holder dead"`.
  > 2. **Free, or holder dead → run it.** Just run `GATE_CMD` — it self-heals/steals the stale lock. Done.
  > 3. **Held by a live PID → is it LEGITIMATE?** Legitimate = `ppid` ≠ 1 **and** its worktree's branch isn't already merged: `WT="$(lsof -a -d cwd -p "$PID" | tail -1 | awk '{print $NF}')"; git -C "$WT" branch --show-current`. If it's a dead orphan or a merged-branch gate, **stop and hand back naming the holder** — don't wait on it, don't clear it yourself.
  > 4. **Legitimate live holder → wait and retry.** Run `GATE_CMD` as its **own** Bash call, bounded by the **Bash tool's `timeout` parameter (~180000 ms)**. That timeout IS your retry timer — there is NO `timeout`/`gtimeout` binary on macOS, so do NOT shell out to one, and do NOT background/detach the gate. If the window expires with no verdict, go back to step 1 (re-check ownership) and re-issue — repeat. (A killed gate leaves a stale `mkdir` lock, but the next attempt's step-1 check steals it, so this is safe.)
  > 5. **Hard cap: ~8 minutes cumulative.** Track your OWN total wait across all retries and cap regardless of whether the holder still reads as legitimate — the lock is an unfair `mkdir` race with no ordering, so "still legitimate" can be a *different* holder each check. The instant you cross ~8 min, OR you find the holder is dead work, stop riding it: push what you have and open a **DRAFT PR** annotated "gate unverified — lock-contended."

- **No AI attribution.** The harness's default is to add a `Co-Authored-By: Claude …` trailer and a "🤖 Generated with Claude Code" PR line; this workflow overrides that default. Paste this into every brief:

  > No AI attribution: no `Co-Authored-By: Claude` trailer on any commit, no "Generated with Claude Code" line in the PR body. Attribute solely to the configured git user.

### Dispatch in the background, then monitor for divergence
**Default to dispatching implementers in the background** (the Agent tool's `run_in_background: true`) rather than blocking on each one. This keeps you — the orchestrator — responsive: you get an automatic notification when an agent completes or opens its PR, AND you can poll the live worktrees mid-flight to catch a wandering agent *before* it burns a whole run going the wrong way. (Background dispatch is the Agent tool's own flag — this is NOT the banned `isolation: "worktree"`; you still make the worktree yourself via the helper and point a plain agent at it.)

**Poll every ~10 minutes for divergence** while agents run. Use `ScheduleWakeup` to self-pace the checks (≈600s is frequent enough to course-correct early). Don't poll *only* for completion — that arrives as a notification for free; the point of polling is **early divergence detection**, with two cheap riders on the same tick: (a) check whether any slice opened a PR and is ready to review + merge, and (b) **recheck gate-lock availability before releasing the next queued slice**, so you pace dispatch to a freeing lock instead of jamming it (see *On the gate → Pacing with a timer*). Each tick, snapshot what files each agent is actually touching and compare against its assigned scope:

- **Snapshot against the FORK POINT (the merge-base), never HEAD and never the integration tip.** Compute the fork point ONCE per tick — `FP=$(git -C <wt> merge-base HEAD origin/<integration>)` (re-`fetch` first; the tip moves) — then `git -C <wt> --no-pager diff --stat $FP` for the full file scope, plus `git -C <wt> ls-files --others --exclude-standard` for untracked. `git diff --stat $FP` (working tree vs fork point) captures BOTH any already-committed blocks AND uncommitted work-in-progress in one shot — which is what you need, because implementers now **commit LAST** (after simplify + gate), so mid-flight most of the work is still *uncommitted*; diffing the working tree against the fork point sees it regardless of whether it's been committed yet. Two traps, both of which have actually bitten us, so neither is hypothetical:
  - ⚠️ **Don't use `git diff --stat HEAD`** — that shows only *uncommitted* changes, so the moment an agent commits (it commits everything at the end) HEAD goes clean and the diff shows **nothing**, and you'll either think the agent is idle or reach for a range diff against the tip (the next trap). Use `git log --oneline $FP..HEAD` only to *list the commits*, never as your file-scope check.
  - ⚠️ **Don't use `git diff <integration>..HEAD` / `origin/<integration>..HEAD`** (two-dot against the tip). Once another PR merges into the integration branch *while this worktree is open*, that merged work shows up as **phantom additions/deletions** here and triggers a false divergence alarm — the merged PR's unrelated files look like this agent's edits. We hit exactly this: a poll two-dot-diffed against a tip that had just advanced, the unrelated merged PR's files appeared as the agent's, and a **perfectly in-scope agent was wrongly `TaskStop`ped** (then had to be resumed). The fork point never moves — anchor every scope check to `$FP`, same as the PR-review diff (below).
- **What counts as divergence:** editing another task's / another phase's **core source files** (e.g. a "types only" task editing the resolver, or a backend task building UI). **What does NOT count:** compile-driven ripples from the task's own change — exhaustive `switch`/enum/interface/config entries a new member forces, a one-line badge to keep an exhaustive UI match compiling, and `+1`-line fixture edits across many `*.test.ts` files (even ones in another area) from adding a field to a shared type. Spell out this divergence/not-divergence line in the wakeup prompt so the check is mechanical, not a re-judgement each time.
- **Check substance, not just the file count.** Staying in scope ≠ doing it right — a diffstat tells you *where* the agent is editing, not *whether the approach is correct*. On any slice with real design subtlety, periodically **read the actual diff of the 2–3 highest-risk files** (`git -C <wt> --no-pager diff $FP -- <file>`, anchored to the same fork point as the diffstat — NOT `HEAD`, which would omit the committed blocks), not just `--stat`, and confirm the agent is following the intended approach (or genuinely improving on it). **Grep the diff for the specific anti-patterns the brief banned** (e.g. a forbidden call leaking into the wrong layer: `git -C <wt> diff $FP -- <files> | grep -nE '<bannedSymbols>'`). A wrong-but-in-scope approach is exactly what file-counting misses, and catching it mid-flight saves a whole wasted run. Keep it light — the risk files, not the entire diff.
- **If an agent diverged** (out of scope OR on a wrong approach): `TaskStop` it immediately, then **dispatch a fresh plain agent into the SAME worktree path** with a tighter brief — explicit *do-not-touch* boundaries naming the files/areas it strayed into (or the wrong pattern to avoid), and an instruction to revert the bad edits. Don't try to course-correct a still-running agent mid-stream; stop-and-replace is cleaner.
- **If an agent opened its PR or completed:** stop rescheduling for that one and move into the PR review loop + merge below.

Carry forward a one-line note of each agent's last-known on-scope file set in the next wakeup prompt, so a sudden jump in surface area is obvious tick-over-tick.

### The PR review loop
Each implementer opens **its own PR** back to the integration branch. You then **review each PR — and actually read the code involved**, not just the agent's summary or the green gate. Read the diff: verify correctness, that it does what the issue/plan intended, that the scope is right, and scrutinize anything the agent flagged or where it went wider than the planned files. A green gate and a confident agent report are necessary but **not sufficient** — the merge decision is yours and must be grounded in the actual changed code.

**Review the implications, not just the brief.** A PR can match its slice's brief, pass its own gate, and stay perfectly in scope — and still be wrong once it lands among its siblings. Read for *second-order* effects, not only local correctness: (1) **cross-slice composition** — when two slices each own one half of a contract (slice A stamps a header, slice B reads that header for the credential; one slice renames/derives a field, another consumes it), confirm the halves still agree *on the merged branch*, even though the files are disjoint. (2) **shared global state** — a change that's correct in isolation can be silently defeated by a process-wide cache, singleton, or memo it didn't author (does that cache key on the value you just made per-tenant, or globally?). (3) **emergent interactions** — a fix that corrects one path can expose or shift a neighbor's behavior (tightening identity makes a queue correctly *wait* instead of mis-running; a now-required field surfaces a caller nobody listed). File-disjointness buys *textual* independence, never *semantic* independence — so the review asks "what does this imply for everything it touches indirectly?", not just "does it match the brief?". When a second-order effect is load-bearing, go read the shared dependency yourself (the cache, the consumer, the sibling's half) rather than assuming it composes.

**Diff against the merge-base, not the integration tip.** The integration branch advances *while a worktree is open* (other agents' PRs merging in parallel), so `git diff <integration-branch>..HEAD` shows the OTHER work's additions as phantom **deletions** in this PR — a clean change then looks like it ripped out unrelated code, and you'll raise a false alarm. Always diff the fork point: `git -C <wt> diff $(git merge-base HEAD origin/<integration>)..HEAD` (re-`fetch` first — the tip moves). Confirm by test-merging the current base in: if the "deleted" files come back as *additions*, the PR never touched them. This bit us once — burned a review cycle chasing a "destructive deletion" that was just the base having moved.

If it needs changes, **dispatch a fix agent into that same worktree**; iterate until the PR is good. Only when satisfied do you **merge and clean up** — and the *order* matters because of worktrees (see next section): the implementer has already pushed everything, so you **remove the worktree first**, then merge with `gh pr merge --merge --delete-branch`.

### Merging a shared hotspot — `mergeable` ≠ correct
When several slices edited the **same function or structure** (a control loop's `tick()`, a reducer, a registry, an exhaustive switch — anything the decomposition flagged as a *shared hotspot*), a clean auto-merge is **not** proof of correctness. Git merges by line: two slices that added code in *different* spots produce `mergeable: MERGEABLE` with no conflict markers — but git has no idea the two additions must **compose**. The recurring failure shape: a structure has sibling branches that must stay **twins** (e.g. two "parked → freeze every kill-clock" branches), each slice added its new clock's freeze-line only to the branches that existed *when it forked*, and the merge silently leaves the newer branch missing it — textually clean, behaviorally a reaper that kills a healthy job. **The integration gate won't catch it** unless a test exercises that exact cross-slice interaction, which is precisely the case nobody wrote.

So whenever a merge — auto-clean OR conflicted — touches a structure 2+ slices edited: **open the merged region and read it; do not trust `mergeable`.** Diff the parallel-edited siblings against *each other* — whatever should be parallel (twin branches, every case of a switch, all handlers of an event, the reducer vs the renderer) must have received the **same** treatment from each slice. Honor the **shared invariant** the decomposition handed you, if any. Fix drift forward as a small, clearly-messaged merge-reconcile commit, then re-verify the interaction (targeted tests, or the full integration gate once the fleet is quiet). Resolve actual conflicts by **merging the base into the branch in its worktree** (`git fetch && git merge origin/<integration>`, keep BOTH concerns) — never rebase — then push and `gh pr merge`. This bit us twice in one governor epic: two separate textually-clean auto-merges each dropped a parked-state freeze-line, either of which would have reaped a healthy queued job.

### The canonical gate command — run `GATE_CMD` (locked), never a bare test runner
There is ONE correct way to run tests in this repo, and improvising around it is what burns hours. Encode it mechanically:

- **Run `GATE_CMD` itself — exactly one, in the foreground, with the exit code captured INLINE.** For Trinity that's `pnpm gate` (= `with-test-lock.mjs sh -c "pnpm format && pnpm check && pnpm test"`). The `with-test-lock` wrapper polls-and-races the `mkdir` mutex, acquires on its own, then runs. Launch one with the full Bash timeout and let it sit (`waiting for test lock held by pid N` → `test lock acquired` → runs). **Always append the exit-code echo so the SINGLE run gives you everything you need: `pnpm gate; echo "GATE EXIT: $?"`** (the foreground run already streams the full format/check/test output to you; the trailing `echo` hands you the verdict in the same call). That one invocation yields BOTH the output and the pass/fail status — so there is **never** a reason to fire a second `pnpm gate` "just to confirm the exit code." A gate that streamed all packages green IS green; re-running the whole locked suite to read `$?` is minutes of compute for a number you already had. Do NOT re-issue it (re-racing just adds a racer and can starve you), and do NOT background-and-yield (that orphans the lock and strands the work — the only sanctioned non-foreground path is the harness `run_in_background` flag treated as a resume signal that *immediately* pushes the PR on completion).
- **⛔ Run `GATE_CMD` from the WORKTREE ROOT — a "missing" gate means you're in a subpackage, not that the gate doesn't exist.** `GATE_CMD` (`pnpm gate`) and the `scripts/with-test-lock.mjs` it wraps live at the **monorepo root** (the worktree top level), NOT in any package. Trinity is a monorepo: when you `cd` into `trinity/`, `cf/`, `trinityailabs.com/`, or `shared/` to do your work, `pnpm gate` reports "no such script" and `scripts/with-test-lock.mjs` looks absent — because you're one level too deep, not because they're gone. The fix is **`cd` up to the worktree root and run `pnpm gate` there**. It is NEVER "the gate isn't a separate script, so I'll just run `pnpm format && pnpm check && pnpm test` myself" — decomposing the gate that way runs the test phase OUTSIDE the machine-wide lock (the very bypass the next bullet bans) and defeats the entire point of `GATE_CMD`. Gate-not-found ⇒ wrong directory ⇒ go up; it is never licence to improvise.
- **⛔ NEVER substitute a bare test runner FOR THE GATE — but scoped/isolated runs DURING dev are fine.** The ban is narrow: do not let a bare runner *stand in for* the locked wrap-up `GATE_CMD`, and do not fire a *heavy* bare run concurrently with the fleet's gates. `vitest run`, `pnpm --filter <pkg> test`, `turbo test`, `pnpm test --filter …` run the tests **OUTSIDE the lock** — only the root `test` script is wrapped by `with-test-lock`, and a package's own `test` script (e.g. `trinity` → `vitest run`) is bare. Running one *as your conclusion* bypasses the mutex entirely and (under contention) saturates the box next to any other gate (parallel vitest forks + workerd) — the exact thing the lock exists to prevent. What this does **NOT** ban: iterating with cheap/scoped/isolated checks WHILE you work — a single targeted test file, a package-scoped `check`/`typecheck`/`lint`/`format` — outside the lock to go fast (this is the *cheap-checks-during* rule in the Implementer section, and it's encouraged). The line is purpose, not command: a scoped run to check your own work mid-task is fine; a scoped run *in place of* the final locked `GATE_CMD` is not. Always CONCLUDE with the one locked `GATE_CMD` from the worktree root. (`check`/`typecheck`/`lint`/`format` are unlocked `turbo` tasks — fine to run alone, but they are NOT the test gate.)
- **To run the locked tests scoped to one package** (e.g. to skip an unrelated package's known-flaky failure), keep it INSIDE the lock: `node scripts/with-test-lock.mjs turbo test --filter=<pkg> --concurrency=1`. Wrapping the runner in `with-test-lock` is the ONLY correct way to scope; a bare `pnpm --filter <pkg> test` is the bypass trap above.
- **No queue ⇒ run when you can actually win.** The mutex has no ordering (see *Implementer* lock mechanics): a gate launched under contention races every other waiter each time the holder frees it and can lose indefinitely. So check the lock first (top hard-rule), and the orchestrator's integration/backstop gate runs only when the box is QUIET; an implementer rides bounded ~3-minute check→wait→check gate windows only behind a single *legitimate* live holder, never into a herd.

### On the gate — re-gate as a backstop, not a blanket
**The IMPLEMENTER runs the project's `GATE_CMD` to green at wrap-up — that IS the gate.** `GATE_CMD` (e.g. `pnpm gate`) is the HEAVY full suite: typecheck + build + lint + the entire test run, behind a machine-wide lock that exists specifically because running it concurrently saturates the box (parallel vite builds + vitest forks). An agentic pre-commit hook may be wired (PreToolUse → gate, denies on failure), but it did NOT intercept the commits observed in orchestration sessions — a broken-test commit slipped through, and a sub-agent reported it never fired. Likely because implementers run as Agent-tool sub-agents and the interactive `PreToolUse` hook doesn't apply to their Bash calls (it gates a normal interactive session as designed — don't assume it's broken there). And release-branch PRs typically get no CI. So: **don't assume a commit was gated by the hook — especially a sub-agent's**; the implementer's own gate run is what stands in for CI.

**The orchestrator must NEVER pile an extra full-gate run on top of a running fleet.** Launching your own `GATE_CMD` while implementer agents are active is the exact saturation the machine-wide lock is there to prevent — every concurrent heavy gate serializes on the lock and starves the agents. The implementers' lock-serialized wrap-up gate runs ARE the per-PR signal; don't duplicate them. Any orchestrator backstop gate must wait until the fleet is QUIET (≤1 agent active).

**Clearing a strangling lock holder is the orchestrator's job — and only when it's dead work.** A *dead* holder self-heals (the next waiter steals it within a poll), so you never touch those. But a **live orphan** (the lock's `pid` is alive yet its `ppid` is 1 — a detached gate whose agent is gone) never releases on its own and never gets stolen, and a holder whose **branch already merged** is running a moot gate; either one strangles the whole fleet indefinitely. When an implementer hands back "I'm waiting on holder pid X / worktree Y / branch Z" — or you spot one yourself (`ps -p "$(cat "$LOCK/pid")" -o pid=,ppid=,command=`, then `lsof -a -d cwd -p <pid>` for the worktree, then check whether that branch is merged) — confirm it's genuinely dead work, then **clear it via `~/.worktrees/remove-worktree.sh <leaf>`**, which kills every process rooted in that worktree (releasing the lock cleanly) and removes it. **Never `rm` the `$LOCK` dir to unstick the fleet** — that races a possibly-live holder and corrupts its run; always go through the worktree helper so the holder process is actually signaled, not just the lock file deleted.

**Per-PR, the orchestrator reads the diff — it does NOT re-run the heavy gate.** Don't blanket re-run the full suite on every PR — it's redundant when the agent credibly reported all-green with real pass-counts. Re-run `GATE_CMD` as a **backstop** only when there's genuine reason: an agent died/stopped early or gave no credible pass-count (gate may never have run), or a **merge integrates branches that weren't tested together** (run one integration gate on the merged result, once the fleet is quiet). Even then, **prefer the minimum**: run only the affected/changed test files rather than the whole suite when possible. Your always-on job is reading the **diff**; heavy re-gating is conditional and fleet-quiet.

**Wide independent fan-outs — DEFAULT to deferred-gate AT DISPATCH; don't serialize N full gates on one lock.** This is a decision you make WHEN YOU FAN OUT, baked into every brief from the first dispatch — NOT a "fallback if it gets contended" you remember to trigger mid-flight once agents are already stranding. The instant you're dispatching N independent slices you'll diff-review anyway, the default is option (2) below: each pushes after a scoped check and you run ONE integration gate at the end. When you fan out many *independent, individually diff-reviewed* slices that all finish coding around the same time, having each run the full locked `GATE_CMD` serializes them all on the one lock: it admits one holder at a time while the rest poll-and-race for it (no queue, no place in line — see the implementer's lock mechanics below), so per-gate lock waits balloon past the implementers' Bash-call timeouts, and agents start detaching their gates and stranding committed-but-unpushed branches (observed: a fan-out where every slice sat waiting on the lock 10–17 min and several stopped with no PR). Avoid it, in preference order: (1) **cap how many implementers gate concurrently** — dispatch in smaller waves, or tell later slices to hold their gate until the lock is free; or (2) for slices you are confident are independent and whose diffs you will read anyway, have implementers **push without the full gate** (a fast scoped check only — e.g. typecheck of their own package) and run **one locked integration gate yourself at epic completion**, on the merged result, once the fleet is quiet. The full per-PR gate earns its keep when slices are coupled or you can't review every diff; for a clean independent fan-out it is N× the cost for the same signal and it is precisely what creates the pileup. Whichever you pick, say so explicitly in each brief so implementers know whether the full gate is their wrap-up signal or not.

**Pacing with a timer — recheck lock availability, don't guess.** When you hold queued slices back to keep from jamming the lock, make the release decision concrete instead of a vibe: on each `ScheduleWakeup` tick (the *same* timer you already run for divergence), read the gate lock before dispatching the next slice — the holder PID file plus the live `with-test-lock` count (the diagnostic in *Implementer* below) — and release the next implementer only when the lock is free or down to a single *legitimate* holder (not a strangling orphan / merged-branch gate, which you clear instead). A `ScheduleWakeup` of ~600–1200s is a fine cadence; a slice's own completion notification re-invokes you sooner the moment it frees a slot, so the timer is the backstop, not the primary clock. This matters because the alternatives both fail under contention: fire every queued slice at once and you get the thundering herd (N gates racing one lock, agents stranding); wait *only* on notifications and a stranded agent — one that stopped at the lock without opening a PR — never sends one, so the loop dies silently. The timer is what makes "pace to the lock" a repeatable check and guarantees you come back to drive the merge-and-dispatch loop even when no notification fires.

(Observed: agents can die mid-run on API/socket errors or stop early — the ones that had already opened a PR were reviewable/mergeable; one that stopped with no PR left only a dirty worktree, forcing manual cleanup. That's why the stop-and-report rule is mandatory in every brief.)

### Transient-red window (schema-first / all-or-nothing epics)
When a foundational change lands first — a NOT-NULL schema swap, a required interface field, a renamed module — it makes `GATE_CMD` impossible to get fully green branch-wide until every consumer migrates. A full-green gate is **unattainable mid-epic BY DESIGN**; holding implementers to it just paralyzes progress.

**In that window the bar shifts — but the COMMAND does not.** The implementer still runs the LOCKED `GATE_CMD`. **Never substitute a bare `check` / `typecheck` / `lint` / `build`** to "go lighter": those compile steps run OUTSIDE the machine-wide lock (only `GATE_CMD` and the test step take it), so a heavy `tsc`-across-the-monorepo launched standalone in parallel across worktrees is the exact box-saturation the lock exists to prevent. (The "heavy" in the gate is the test suite *and* the typecheck — a bare `check` is not the light option it looks like, and it's unlocked.) What changes is how you READ the gate's result: `GATE_CMD` exits non-zero on the pre-existing baseline failures until every consumer migrates, and that's expected, **not** a stop. The pass bar becomes:
- `GATE_CMD` actually ran, and its compile half (typecheck + lint) is GREEN.
- The task's OWN new/affected tests are GREEN in the output.
- NO failures beyond the baseline SET that pre-dated this task **on its fork point** — file + test names, not a count (fixing one stale test and breaking a new one keeps the count identical but is still a regression). The absolute count drifts between tasks forked at different tips and as consumers migrate; only the per-fork SET is meaningful.

Don't chase a green exit, and don't loop the gate.

**Who runs the full gate, and when.** The implementer runs `GATE_CMD` **once** at wrap-up — that is its signal. The ORCHESTRATOR does **not** add its own `GATE_CMD` runs on top of a live fleet (the saturation rule), and runs a full confidence gate only at **quiet-branch checkpoints** (fleet idle, no open PRs) and the **epic-completion sign-off** — once the baseline failures return to zero, `GATE_CMD` must be fully GREEN on a quiet branch; that is the definitive gate before the work flows onward.

**N green PRs ≠ a green epic — gate the integrated whole before you close.** Each branch ran its gate against *its own fork point*, blind to the other slices; the merged integration branch with all of them together has been gated by *no one*. So a multi-slice epic is NOT done when every PR is individually merged-green — the close-out is: once the last slice merges and the fleet is quiet, run ONE `GATE_CMD` on the integration branch itself and require it GREEN (modulo a known pre-existing baseline) BEFORE declaring the work done or closing the issue. A failure that first appears here is a real cross-slice interaction the per-PR gates *structurally could not* catch (none of them saw the others) — fix it forward, don't rationalize it away. Same principle as re-reading a shared-hotspot merge: independent-green is necessary, *integrated*-green is what actually ships.

### Merge & cleanup — order matters because of worktrees
The branch lives in a linked worktree, and **git refuses to delete a branch that's checked out in a worktree** (`cannot delete branch 'X' used by worktree …`). So `gh pr merge --delete-branch` would error on the local-branch step if the worktree still exists. The fix is ordering, not dropping the flag: the implementer has already pushed everything and the PR is up, so **tear the worktree down before you merge**, and then `--delete-branch` cleanly removes both branches. Sequence, every time:
1. **Remove the worktree first — always via the helper:** `~/.worktrees/remove-worktree.sh <branch-leaf>` (or the absolute path). **Do not use `git worktree remove` directly.** Plain `git worktree remove` evicts the directory but does NOT signal any running processes — so a wrap-up `pnpm gate` that the implementer agent detached or backgrounded survives as an orphan, keeps holding the machine-wide lock, and starves the next agent racing for it. The helper kills all processes rooted in the worktree first (via `lsof` + `pgrep`), escalating SIGTERM → SIGKILL, then removes the worktree. Nothing is lost — it's all pushed — and this frees the branch so it can be deleted.
2. **Merge with branch deletion:** `gh pr merge --merge --delete-branch` — real merge commit (never squash), and with the worktree gone `--delete-branch` cleanly removes BOTH the local and remote branch. Skipping `--delete-branch` is what let merged `origin/*` branches pile up indefinitely (they did: dozens accumulated). Verify it's fully merged first; STOP on git's "not yet merged to HEAD" warning.
3. **Sync the local integration branch:** `git checkout <integration> && git pull --prune` so it fast-forwards to the merge commit and stale remote-tracking refs are pruned. Skipping the sync leaves it behind the remote, so the next worktree gets cut from a stale HEAD.
4. **Close the issue yourself if the work resolved one — GitHub will NOT auto-close it.** Auto-close only fires when a PR with a `Closes #N` keyword merges into the **default branch** (`main`). Our PRs merge into the **integration branch**, so a linked issue stays open regardless of keywords — and in practice the PRs only cross-reference (`#N`) rather than use a closing keyword anyway, so no close link even exists. After merging, `gh issue close <N> --comment "Fixed in #<pr> (merged into <integration>)."` for each issue the merged work resolved. (Don't wait for the eventual integration→`main` merge to sweep them — close them now so the issue tracker reflects reality.)

## Implementer

Work only in the assigned worktree (absolute paths), match surrounding style, and follow the project's conventions — its `BRIEF_CONVENTIONS` (from `~/.worktrees/config/<project>.sh`) and its `AGENTS.md`. (For Trinity — **app = Solid, sidecar = Effect**: invoke the `effect` skill for any Effect-TS code on the sidecar, and the `solid` skill for any SolidJS UI code on the app — `solid-js`, `@solidjs/router`, `@tanstack/solid-query`, Kobalte; build forward-only — pre-launch, no backwards-compat shims; comments explain the mechanism, no issue/PR/version/plan refs.)

**The flow is strictly ordered: write all the code → `/simplify` → `GATE_CMD` green → commit LAST.** Do NOT commit as you go. Write the whole change in your worktree (running *cheap* checks as you work is good — format, a scoped lint/typecheck of the file or package you're in, a single targeted test file; just don't fire the heavy locked `GATE_CMD` repeatedly). When the work is done, run the `/simplify` skill over your *full uncommitted* working-tree diff (reuse / simplification / efficiency / altitude cleanups — quality only, not bug-hunting) and fold the fixes in. Then run the project's `GATE_CMD` **once** and make it green. ONLY then do you commit.

⚠️ **Why commit last:** `/simplify` acts on the *uncommitted* working-tree diff. Anything you've already committed is invisible to it — so if you commit as you go, the earlier blocks never get simplified, and against a fully-committed (clean) tree `/simplify` sees an empty diff, finds nothing, and silently does nothing while reporting "nothing actionable." The review simply never happens. Keeping everything uncommitted until after simplify + gate is what makes the simplify pass actually cover the whole change.

**Then commit in logical blocks** — committing last loses nothing: stage each self-contained step (a type + its plumbing, a behavior + its tests, one cohesive refactor) into its own commit with a mechanism-explaining message (`git add -p` / path-scoped staging), so the never-squashed history stays reviewable block-by-block — now over code that's already simplified and green. Then push all your commits, open a PR targeting the integration branch, and report back: PR URL, files changed, gate result, tests touched, and anything ambiguous. **Never rebase. Never merge your own PR.**

**⛔ Attribute every commit + PR to the git user ONLY — no Claude attribution, ever.** This OVERRIDES the harness default that tells you to end commit messages with a `Co-Authored-By: Claude …` trailer — that default does NOT apply here. Do NOT add a `Co-Authored-By: Claude …` trailer (any model) to commit messages, and do NOT add a "🤖 Generated with Claude Code" line to PR bodies. Keep messages clean and attributed solely to the configured git user.

**The gate under fleet contention — know how the lock actually works; it is NOT a queue.** `GATE_CMD` runs behind a machine-wide lock, so when several worktrees wrap up together your gate may wait many minutes before it runs. That wait is EXPECTED, not a hang and not a failure. But understand the mechanism so you act on facts, not impatience:

- **It's an `mkdir` mutex, not a fair queue.** The lock is a directory at `$TMPDIR/trinity-test-gate.lock` (the holder's PID is in the `pid` file inside it). `GATE_CMD` is locked because `pnpm test` re-invokes `scripts/with-test-lock.mjs` — a script that lives at the **worktree root** alongside the root `package.json`'s `gate` script, NOT inside any subpackage; if either looks "missing," you're in a subpackage and need to `cd` up to the worktree root (see the canonical gate command above), never to improvise around the gate. The gate's own typecheck/build are NOT locked. Waiters **poll every ~2s and race** to `mkdir` the instant the holder releases — there is no ordering and no fairness, so concurrent waiters are a thundering herd and an unlucky run can be starved. "Queued" is a misnomer; nobody is holding your place in line.
- **Wait behind a legitimate holder in bounded ~3-minute windows, not one all-or-nothing call — but cap the TOTAL wait at ~8 minutes.** One `pnpm gate` call polls and acquires on its own — it prints `waiting for test lock held by pid N`, then `test lock acquired`, then runs. But don't bet the whole wait on a single call at the FULL Bash timeout: a live fleet holder can easily outlast that window, and if it does you've spent the whole call and learned nothing you couldn't have learned in the first few minutes. Instead, launch it with a bounded timeout (~180s / 3min). If that window elapses with no verdict (still `waiting for test lock held by pid N`), that's expected under contention, not a failure — re-run the ownership check below, confirm the holder is still legitimate, and re-issue the gate for another ~3-minute window. **This cannot loop indefinitely: the lock is an unfair `mkdir` race (see below), not a queue, so a "still legitimate" holder can in principle be a different process each time you check — a run of bad luck in that race is real starvation, not a hang, and "still legitimate" alone is not license to keep waiting forever.** So track your OWN cumulative wait across re-issues and hard-cap it at **~8 minutes total** (roughly two to three ~3-minute windows — pad upward, not down, as the suite grows) regardless of what the ownership check says: once you hit that ceiling, stop riding it and fall through to the stuck-handback rule (push + open a DRAFT PR annotated "gate unverified — lock-contended"), the same as if the holder had turned illegitimate. Below the ceiling: check, wait, check again, until either the gate acquires and streams a verdict, or the ownership check flips (the holder dies, orphans, or its branch merges) and you know to stop or hand back instead.
- **Check who owns it — and whether that owner is LEGITIMATE — before you commit to waiting.** A *live* PID is not enough to justify a wait: the holder can be a real gate that will finish and open a PR (ride it), OR dead work you'd wait on forever — a detached orphan whose agent is gone, or a holder whose branch already merged. Diagnose, don't guess:
  ```bash
  LOCK="$(node -e 'process.stdout.write(require("os").tmpdir())')/trinity-test-gate.lock"
  PID="$(cat "$LOCK/pid" 2>/dev/null)"
  ps -p "$PID" -o pid=,ppid=,command= 2>/dev/null || echo "lock free / holder dead"
  WT="$(lsof -a -d cwd -p "$PID" 2>/dev/null | tail -1 | awk '{print $NF}')"  # holder's worktree
  git -C "$WT" branch --show-current                                          # holder's branch
  pgrep -fl with-test-lock   # every gate process on the box
  ```
  Read it as legitimacy, not just liveness:
  - **Free / holder dead** → your run acquires (or steals — see below) immediately; just launch your one gate.
  - **Live, parented to a real process (`ppid` ≠ 1), branch not yet merged** → a legitimate fleet holder. You are genuinely waiting — let your one patient foreground gate ride.
  - **Live but `ppid` is 1 (a detached orphan), OR the holder's branch is already merged** → you're waiting on dead work that will *never* release on its own (an orphan's PID stays alive, so it never self-heals; a merged branch's gate is moot). Do NOT wait it out — but also do NOT `rm` the lock or kill another worktree's process yourself: clearing a sibling worktree is the ORCHESTRATOR's job (it owns the global view), not an implementer's. **Stop and hand back**, naming the strangling holder (its pid + worktree + branch) so the orchestrator can clear it via the helper.
- **Dead holders self-heal — but a LIVE orphan does NOT; don't `rm` the lock either way.** The script checks the holder PID's liveness, so a crashed/SIGKILLed holder's lock is auto-stolen by the next waiter within a poll — a *dead* stale lock is never a permanent block. A *live* orphan (ppid 1) is the opposite: its PID answers `kill -0`, so the script treats it as a valid holder and never steals — that one needs the orchestrator to clear the worktree (above), not a steal. Either way, never manually delete `$LOCK` to "unstick" yourself — at worst you'd nuke a live holder's run. Only consider it if `ps` proves the PID is dead AND stealing demonstrably isn't happening (rare).
- **Never run two gates from your own session at once.** A backgrounded gate (e.g. `pnpm gate | tail`) plus a foreground re-run race each other on this very lock and waste both. Exactly one gate attempt in flight at a time.
- **One run gives you the verdict — never re-gate to read the exit code.** Capture `$?` INLINE on the single run (`pnpm gate; echo "GATE EXIT: $?"`), so the same call that streams the output also hands you pass/fail. A second full `pnpm gate` fired only to "confirm the exit code" after the first already streamed every package green is pure waste — it re-takes the lock and re-runs format + check + the entire suite for a number you already had. If the gate reached the test phase with all packages passing, it passed; read the result you have, don't pay for it twice.

Re-issue the foreground gate on that same **~3-minute check→wait→check cadence, capped at ~8 minutes of total wait**: each bounded window ending with no verdict — whether from the timeout you set or the harness's own Bash-call limit killing the poller — is expected under contention, not a failure, UP TO that cap. Before every re-issue, re-run the ownership check above so you know whether you're still riding a live legitimate holder or free to acquire. Do NOT route around the wait: **never background/detach the gate and then stop**, reporting "I'll wait for a notification." A detached gate becomes an orphan that keeps holding the lock and starves the next agent, and YOU are never re-invoked — so the PR never opens and your committed work is stranded (this has repeatedly left whole fan-outs of committed-but-unpushed branches with no PRs). The ONLY sanctioned non-foreground path is the harness `run_in_background` flag, and only if you treat the harness completion re-invocation as your resume signal and **immediately push + open the PR** when it fires; stopping "done" with a raw detached pid and no PR is never acceptable. **"Genuine waiting" is bounded, not open-ended: once your cumulative wait crosses ~8 minutes — even if the holder still reads as legitimate — stop and fall through to the stuck-handback rule below.** The lock's poll-and-race mechanics (no ordering, no fairness) mean an unlucky implementer can keep losing the acquire race to a rotating cast of "legitimate" holders indefinitely; the 8-minute cap is what turns that open-ended starvation risk into a bounded one. Push and open a DRAFT PR — never a committed-but-unpushed branch with no PR. (Note: if the orchestrator's brief told you this is a capped/integration-gated fan-out where you should push *without* the full gate, follow that instead — run only the scoped check it named.)

**Don't be lazy — leave the tree cleaner than you found it, and commit out-of-scope fixes separately.** Your job is not "make the assigned change and stop the instant the gate stops *erroring*." A gate that exits green but still prints warnings is NOT done. Actively fix every lint warning, type warning, and error you hit or that your own change surfaces — even when it is technically out of scope: the residual unused imports a file is left with after you move its code to a sibling, a path-keyed lint allowlist (sanctioned-git / ambient-identity / ambient-network rules) that must follow the relocated code to its NEW file path, a stray nit in a file you only opened to read. The bar is **zero warnings and zero errors on every file your change touches.** **Isolate each such opportunistic / out-of-scope fix in its OWN commit** with a clear mechanism-explaining message (e.g. a `chore(lint): prune imports the split left unused` commit, kept separate from the `refactor: split …` commits) — so it stays independently reviewable and the orchestrator can keep or drop it without untangling it from the task. Never silently fold an unrelated fix into a task commit, and never hand back a "green" PR that still carries warnings for someone else to clean up. (If a surfaced fix is genuinely large, risky, or a behavior change, still isolate it in its own commit AND call it out in your report — don't bury it, and don't skip it.)

**If you get stuck, hand back a reviewable artifact — never spin or die silently.** A blocker, an ambiguity you can't resolve, a gate you can't get green, or running low on room all mean the same thing: STOP and give the orchestrator something it can act on. Commit-push and open a **draft PR** with what you have; if you can't even do that, hand back a complete report of your worktree state — what's done, what's broken, the exact error/blocker, and the decision you need. Partial-but-clean beats stuck-and-silent (e.g. land the solid part and flag the rest). **Never leave a dirty worktree with no PR and no report — and never leave a committed-but-unpushed branch with no PR; if your gate won't clear (e.g. lock contention), push and open a DRAFT PR annotated "gate unverified — lock-contended" so the orchestrator inherits a reviewable, mergeable artifact rather than invisible work trapped in your worktree.**

## Hard rules (both roles)

- **⛔ Read the gate lock BEFORE launching ANY gate — never fire blind.** The gate's test phase takes a machine-wide `mkdir`-mutex that is NOT a queue — waiters poll-and-race, no place in line (see *Implementer* lock mechanics). So before *anyone* starts a gate — an implementer's wrap-up gate OR an orchestrator's backstop / integration gate — run the ownership check FIRST: `pgrep -fl with-test-lock` for every gate on the box, plus the holder's `pid`/`ppid`/branch via the diagnostic in *Implementer*. Firing without looking means you may launch a gate that just sits racing a lock you never inspected — **a ticket to a closed door** — when the holder might be dead work you should CLEAR (a detached orphan with `ppid==1`, or a gate whose branch already merged) or a legitimate live fleet gate you should PACE BEHIND, not pile onto. Read the lock, THEN choose: acquire (lock free, or a dead holder self-heals on the next poll), ride one legitimate live holder with bounded ~3-minute check→wait→check gate windows, or clear an orphan/merged-branch holder via `~/.worktrees/remove-worktree.sh <leaf>` (NEVER `rm` the lock dir). A clean acquire you got by luck of timing is not the same as one you earned by checking — check every time.
- **⛔ Never use the Agent tool's `isolation: "worktree"` param or any auto worktree provisioner.** Make worktrees only via `~/.worktrees/setup-worktree.sh`, verify HEAD, then dispatch a plain agent. (See worktree section.)
- **Never squash-merge.** Merge PRs with a real merge commit (`gh pr merge --merge --delete-branch`), never `--squash` — squashing flattens the individual commits we keep. Always pass `--delete-branch` so the remote branch is removed on merge instead of accumulating — but **remove the worktree first** (see Merge & cleanup), or `--delete-branch` errors trying to delete a branch still checked out in a worktree.
- **Do NOT rebase.** Rebasing rewrites history. Default to plain merge commits everywhere: integrating parallel worktree branches, a branch falling behind the integration branch, resolving overlap between two PRs — MERGE, never rebase. If two parallel branches touch the same file, resolve it at MERGE time (merge one, then merge the other and fix conflicts in the merge), not by rebasing one onto the other. Never instruct a sub-agent to rebase. Only rebase when there's a specific, unavoidable reason AND it's worth rewriting history — rare; when in doubt, merge.
- **⛔ `/simplify` only acts on UNCOMMITTED changes — so commit LAST.** It reads the working-tree diff, so against a clean (fully-committed) tree it silently no-ops — finding "nothing actionable" because there is nothing to see, not because the code is clean. The order is load-bearing: write all the code → `/simplify` → `GATE_CMD` green → *then* commit (in logical blocks). Never commit as you go and never commit-everything-then-simplify — either one hides code from the simplify pass and the review never happens. Cheap checks (format, scoped lint/typecheck, a single test file) may run mid-task; the one heavy `GATE_CMD` runs at the end, after simplify, before the commit.
- **⛔ No Claude attribution on commits or PRs — git user only.** This OVERRIDES the harness's built-in "end commit messages with `Co-Authored-By: Claude …`" instruction; it does not apply in this flow. Never add a `Co-Authored-By: Claude …` trailer (any model) to a commit, nor a "🤖 Generated with Claude Code" line to a PR body. Attribute solely to the configured git user. Spell this out in every implementer brief — dispatched sub-agents follow the harness default unless told otherwise.
- **Branch from the active integration branch, not `main`.** PRs target the integration branch.
- **Never delete a branch past git's "not merged" warning** — verify fully merged into the target first.

**Why this shape:** integration branch as the merge point, isolated worktrees per task, clean up after merge. Worktrees start without env files and can drift from the integration HEAD, which breaks the work if not handled up front. We preserve real merge history rather than flattening it via squash or rebase. Skipping the post-merge local sync leaves the local integration branch behind the remote, so the next worktree gets cut from a stale HEAD.
