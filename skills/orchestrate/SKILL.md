# Orchestrate — release-branch worktree workflow

Our main dev workflow: we work off an integration branch (for Trinity, `release/x.x.x` — discover the active one with `git branch --list 'release/*'` / the current branch; never hardcode a version). For each task we cut a git worktree, do the work there, commit, push, and open a PR back into the integration branch. The heavy build+test gate does NOT run in the worktree that produced the change — the implementer **enqueues** it and durable orchestrators **drain** it (see *The durable gate queue* below). After a PR merges we **sync the local integration branch first**, then clean up the merged branch and its worktree.

This is project-agnostic. The per-project specifics — which env files to symlink, the install command, the gate command, the docs fast-path, and the conventions to bake into briefs — live in a config the tooling reads (see *Per-project config* below), not in this playbook.

**Default to planning for parallelization.** Given a batch of work, the first move is to decompose it into independent tasks that can run concurrently in their own worktrees — don't default to doing everything sequentially yourself. Spin up as many worktrees/branches as the work needs. Because implementers don't run the heavy gate themselves (they enqueue it), a wide fan-out doesn't serialize on a gate lock — fan out as wide as the work is genuinely independent.

## First: which role are you?

Before acting, decide whether you are the **orchestrator** or an **implementer** — they behave very differently.

- **ORCHESTRATOR** when the user asks you to *orchestrate / coordinate* work, to *complete or work a GitHub issue*, to *execute a plan*, or hands you a batch of tasks. You are the coordinator. **Do NOT write the implementation yourself.** You decompose the work, create + verify worktrees, dispatch an implementer sub-agent into each (in parallel when independent), then run the review loop, **drain the gate queue**, and merge. You hold the plan, the reviews, the gate runs, and the merges — not the file edits.

- **IMPLEMENTER** when you are a dispatched sub-agent, or the user *directly tells you to implement / build / fix* a specific thing. You do the actual code in your assigned worktree, pass a cheap scoped check, push, open a **draft** PR, **enqueue the gate**, and hand back. **You do not run the full gate and you do not merge your own PR** — a runner gates it, and the orchestrator reviews and merges.

## Per-project config

The worktree tooling is generic; each project declares its specifics in a bash file at `~/.worktrees/config/<project>.sh`, where `<project>` is the repo's directory name. The helper sources it; the orchestrator reads it too (e.g. `cat ~/.worktrees/config/<project>.sh`) to learn the gate and the brief conventions. Keys:

- `ENV_FILES` — gitignored env files to symlink from the main checkout into each worktree.
- `INSTALL_CMD` — install command run inside a new worktree (worktrees don't share `node_modules`).
- `GATE_CMD` — the heavy full gate (build + test). This is what the RUNNER runs against a queued PR's worktree; implementers do NOT run it (they enqueue it). It's also what a human runs for a manual full check.
- `SCOPED_CHECK_CMD` — the cheap, unlocked check an implementer's commits are held to (format-check + lint + typecheck; no build, no test). For Trinity this is `pnpm check`, and the agentic pre-commit hook already enforces it on every commit.
- `ENQUEUE_CMD` / `DRAIN_CMD` — how an implementer enqueues its PR for gating, and how the orchestrator drains the queue. For Trinity: `pnpm gate:enqueue` and `pnpm gate:drain`.
- `DOCS_BRANCH_PREFIX` / `DOCS_NOTE` — branches under this prefix skip env + install (a fast path for deps-free work like markdown).
- `BRIEF_CONVENTIONS` — project conventions to bake into every dispatched implementer brief (framework skills, compat policy, comment style, simplify-then-check, etc.).

The config is sourced bash, so it may also `export` env the tooling needs — most usefully a **shared build cache** for a monorepo (turbo/nx/bazel): `export TURBO_CACHE_DIR="$HOME/.cache/<project>-turbo"` lets a fresh worktree's gate **replay** tasks another checkout already built (near-instant on unchanged packages) instead of running cold every time. A remote cache (a shared HTTP/S3 backend the runner declares in its config — e.g. turbo's `remoteCache.apiUrl`) extends the same replay *across machines and CI*. ⚠️ **A cache var must also live in `~/.zshenv`, NOT `~/.zshrc`.** The gate runner, the `DRAIN_CMD` drain, and dispatched implementer agents all run in **non-interactive** shells, which read `~/.zshenv` only — a var set solely in `~/.zshrc` reaches your interactive terminal but *not* the fleet, so the shared cache silently never applies to gated PRs and every drain runs cold. The config export covers the install step; `~/.zshenv` covers the gate. (This is a real trap we hit: the 287x cache speedup showed up in the terminal but not in a single gated worktree until the export moved to `~/.zshenv`.)

**Go through the task-runner, never around it.** A shared cache only helps commands that actually invoke the runner (turbo/nx/bazel): the project's `SCOPED_CHECK_CMD` and `GATE_CMD` do, by construction. Invoking the underlying binary **directly** — `vitest`, `tsc`, `eslint` — or running a per-package script that shells straight to it (e.g. `pnpm --filter <pkg> test` when that package's `test` script is a bare `vitest run`) **bypasses the cache entirely and always runs cold**, both locally and in the drained gate, and never *populates* the cache for anyone else either. So steer every whole-package / whole-suite run — in this skill's cheap-check guidance, in briefs, and in the project's `AGENTS.md` — to the cached runner command (`turbo run <task> --filter=<pkg>`, or the project's own `SCOPED_CHECK_CMD`). Reserve a direct-binary run for a **single targeted file** (`vitest run path/to/x.test.ts`), where wrapping one file in a whole-task cache entry is pointless overhead. If the project's `AGENTS.md` still documents the raw-binary form as the default for a whole package, that's a leak — fix the doc, don't propagate it into briefs.

A repo with no config still works — the helper makes a bare worktree (no env, no install). To onboard a new project, drop a `~/.worktrees/config/<project>.sh` declaring the keys above.

## Worktree creation (both roles depend on this being done right)

**⛔ HARD BAN — the Agent tool's `isolation: "worktree"` parameter (and ANY harness / auto worktree provisioner) is FORBIDDEN. Never use it.** It (a) creates the worktree under `.claude/worktrees/agent-*` (violates invariant 1 below) and (b) repeatedly seeds it at a STALE base — far behind the integration tip (observed ~1000+ commits behind, a previous-minor-version commit) — which has burned us over and over (agents waste cycles re-basing, or stale code silently leaks in). There is exactly ONE way to make a worktree: the helper script below, then **verify HEAD yourself**, THEN dispatch a plain agent (no `isolation`) pointed at that worktree path.

**Helper script — `~/.worktrees/setup-worktree.sh`** (a symlink into the `worktree-orchestrate` repo at `bin/setup-worktree.sh`, version-controlled alongside this skill and its sibling helpers `merge-pr.sh` / `remove-worktree.sh`; `install.sh` wires up the `~/.worktrees/` symlinks. Edit it directly on `main` — no worktree/PR ceremony needed for these tooling repos). It auto-detects the project and reads its config, so it works the same for any repo.

Usage: `setup-worktree.sh <branch> <base>` — both **required**. Run it from **anywhere inside the target repo** (the root, a subdirectory, a monorepo subpackage, or even a linked worktree — it walks up to the repo's common gitdir). If cwd isn't inside the repo, prefix `REPO=/path/to/repo`.
- `<branch>` = full name of the branch to create (any prefix: `feat/…`, `fix/…`, `refactor/…`, `docs/…`); the worktree dir is named after the segment past the last slash (`feat/toasts-top-right` → `toasts-top-right`).
- `<base>` = the branch to fork from, with **no default** (integration branches roll over often, so a hardcoded default just goes stale); it errors early with a `git fetch` hint if `<base>` isn't a local ref yet.

It creates the worktree at `~/.worktrees/<project>/<branch-leaf>`, symlinks the config's `ENV_FILES`, and runs `INSTALL_CMD` — automating invariants 1, 3, 4 below. A branch under `DOCS_BRANCH_PREFIX` takes the fast path (env + install skipped).

**The four invariants** (the helper covers 1, 3, 4 — step 2 is always on you):
1. The worktree lives under `~/.worktrees/<project>/`, **never** under `.claude/worktrees`.
2. **Verify its HEAD == the base tip BEFORE dispatching any agent** — run `git -C <worktree> rev-parse HEAD` and confirm it equals `git rev-parse origin/<base>` (e.g. `origin/release/x.x.x`). The helper does **not** verify this for you — sync the base first, then check. A mismatch means STOP and fix the base; do not dispatch.
3. Env files are symlinked into the worktree (they aren't carried over automatically).
4. Deps are installed (worktrees don't share `node_modules`, and the gate can't run without them).

## The durable gate queue

The heavy gate (`GATE_CMD` = build + full test suite) is expensive and CPU-saturating: running several concurrently across worktrees stacks builds and vitest/workerd forks until the box thrashes. So implementers never run it — they enqueue a durable ticket and hand back, and orchestrators drain the queue and gate one at a time. This keeps the wait off the implementer entirely and makes a mid-flight death impossible to lose work to:

- **Implementers never run the full gate.** They get their code to a cheap **scoped check** (format-check + lint + typecheck; the pre-commit hook enforces this on every commit), commit, push, open a **draft PR**, then **enqueue** a durable gate request (`ENQUEUE_CMD`) and hand back. The work is pushed and the PR is open before the ticket exists, so an implementer death after enqueue strands nothing — worst case the work sits as a visible draft PR.
- **Orchestrators drain the queue.** On each tick a runner (`DRAIN_CMD`) claims a queued ticket, runs `GATE_CMD` **one at a time** in that ticket's worktree behind a slim machine-wide slot, then flips the PR ready (green) or comments the failure and leaves it draft (red). Multiple orchestrators can drain the same queue safely: every state transition is an atomic rename that exactly one runner wins, and a runner that dies mid-gate has its in-flight ticket reclaimed and re-gated (the gate is idempotent).

The consequences ripple through everything below: there's no gate lock for implementers to wait on, no thundering herd of concurrent gates, and no way for a dying agent to strand committed work. Fan out freely; drain steadily; the queue + draft PRs make lost work structurally impossible.

## Orchestrator

### Dispatch
Decompose the batch into independent tasks. For each: create + **verify** a worktree (above), then **dispatch a plain implementer sub-agent** (no `isolation`) pointed at that worktree path — in parallel whenever the tasks are independent. Spin up as many worktrees/branches as the work needs; the gate doesn't bottleneck fan-out (implementers enqueue rather than gate).

**Pick the model tier per task — match cost to difficulty.** Pass `model: "sonnet"` to the Agent tool for the *bulk* of implementer dispatches: well-scoped, mechanical, or moderate work (wiring, UI from a clear spec, a route/op that mirrors an existing one, tests, a refactor with a known shape). Reserve `model: "opus"` for genuinely HARD slices: subtle algorithms, ambiguous or design-heavy work, tricky concurrency/lifecycle, security-sensitive code, or large cross-cutting changes where a wrong approach is expensive to unwind. When unsure, **start Sonnet** — a Sonnet PR you have to redirect is cheaper than Opus on routine work; escalate to Opus only if the task proves harder than scoped. Trade-off: the easier the model, the **more explicit the brief** must be — exact files, patterns to copy, hard do-not-touch boundaries, research-first steps. Clarity substitutes for model strength, so a Sonnet brief should read as near-deterministic steps, not "figure it out."

**Open every brief with "Step 0: invoke the `orchestrate` skill (via the Skill tool) and act as the IMPLEMENTER."** The agent then reads the generic implementer playbook from the *Implementer* section itself — scoped-check-not-full-gate, cheap-checks-during, the zero-warnings "don't be lazy" bar, push-draft-PR-enqueue-report, never-rebase / never-self-merge, and stop-and-report. **Do NOT re-paste any of that into the brief** — it's already in the skill the agent just loaded, and duplicating it bloats the brief and drifts out of sync.

**This skill is written for you, the orchestrator — you are the one who has fully internalized it, and your job is to translate the parts implementers must execute correctly into their brief.** Step 0 ("invoke the `orchestrate` skill and act as the IMPLEMENTER") covers simple absolute prohibitions fine — never rebase, never self-merge — an agent either does or doesn't do those. It does NOT cover (a) a multi-step procedure that has one correct order and one wrong, silently-incomplete order with no error to flag the miss, or (b) a rule that overrides a harness-level default the agent would otherwise follow unprompted. Three items have exactly one of those two shapes, and you paste all three into every brief close to verbatim rather than relying on Step 0 to carry them: the **commit-last ordering** and the **push-draft-PR-then-enqueue handoff** (shape a), and **no AI attribution** (shape b). Treat any future addition with either shape the same way.

Your brief carries the **task-specific context the skill can't know**, plus the three items above — keep the rest lean:
- **Step 0 skills.** Invoke `orchestrate` (as implementer), plus any framework reference skill the task touches — name it as an explicit first step, not "use it." For Trinity — **app = Solid, sidecar = Effect**: Effect-TS on the sidecar (services, layers, HTTP routes, typed errors) → "invoke the `effect` skill"; SolidJS UI on the app (`solid-js`, `@solidjs/router`, `@tanstack/solid-query`, Kobalte) → "invoke the `solid` skill"; a full-stack slice → both. Sonnet agents won't reach for these on their own, so spell them out.
- **The task + the worktree.** What to build/fix, the assigned worktree's absolute path, and the concrete files/patterns to copy. The weaker the model, the more explicit — near-deterministic steps for Sonnet, not "figure it out."
- **Scope + hard do-not-touch boundaries.** The other slices'/phases' core files this task must NOT edit. This is the part only you hold from the decomposition; the skill cannot supply it.
- **Project conventions for this slice.** The relevant bits of the project's `BRIEF_CONVENTIONS` (`~/.worktrees/config/<project>.sh`) and `AGENTS.md` — compat policy, comment style. (Trinity: pre-launch forward-only, no backwards-compat shims; comments explain the mechanism.)
- **Gate mode for this slice.** The DEFAULT (say nothing) is: implementer runs only the scoped check, pushes, opens a draft PR, and enqueues — the runner does the authoritative gate. Override ONLY when the slice is foundational/cross-cutting and you want it verified before it enters the queue (or when there's no orchestrator to drain): then tell the implementer to run the full `GATE_CMD` itself and flip its own PR ready, and NOT enqueue.
- **Simplify pass for this slice — your call.** `/simplify` is a real quality pass (reuse / simplification / efficiency / altitude cleanups) that costs a round-trip and only pays off on substantial work — a new module, non-trivial logic, a refactor with structure to tighten. On a trivial slice — a one-line fix, a copy/string tweak, a config bump, a mechanical rename — it burns that round-trip to report "nothing actionable," so it's noise, not safety. YOU hold the decomposition, so YOU decide per slice and say so in the brief: default substantial slices to a simplify pass, and tell trivial ones to skip it. State it explicitly either way — the decision sets the commit ordering below.
- **The commit ordering — set by the simplify decision above.** If the slice runs `/simplify`, commit LAST: `/simplify` acts only on the *uncommitted* working-tree diff, so anything committed early is invisible to the pass — it reports "nothing actionable" against the clean part whether or not that code needed cleanup. If the slice skips simplify, there's nothing to hold the tree open for, so commit in logical blocks as the work lands. Paste the matching block into the brief:

  > **Simplify slice:** Do NOT commit as you go. Write the ENTIRE change in your worktree first — uncommitted. Cheap checks while you work are fine (format, a scoped lint/typecheck, a single targeted test file) but don't commit yet. When the code is done, run `/simplify` over your full uncommitted diff and fold its fixes in. Then run the auto-formatter in WRITE mode (Trinity: `pnpm format`) over the tree, and THEN commit — in logical, self-contained blocks (a type + its plumbing, a behavior + its tests), never one giant commit and never as-you-go.

  > **Skip-simplify slice (trivial change):** Skip `/simplify` — it has nothing to chew on here. Write the change, run the cheap checks, run the auto-formatter in WRITE mode (Trinity: `pnpm format`), and commit in logical, self-contained blocks (never one giant commit). Do NOT run `/simplify` just to tick a box; the orchestrator judged it not worth the round-trip.

  In both cases your commits are held to the cheap SCOPED check by the pre-commit hook (format-check + lint + typecheck — no build, no test suite); you do NOT run the full gate. The hook only *checks* formatting, it does not *fix* it — so run the formatter in write mode right before you commit, or an unformatted diff gets bounced.

- **The push-draft-PR-then-enqueue handoff.** The implementer's wrap-up is a durable handoff, not a wait — this order is load-bearing because it's what makes a mid-flight death lose nothing. Paste this into every brief (unless this slice is in override gate mode, above):

  > After committing: **push your branch, then open a DRAFT PR** targeting the integration branch. THEN enqueue the gate: `ENQUEUE_CMD --branch <yourBranch> --worktree <yourWorktreeAbsPath> --pr-number <n> --pr-url <url>` (get the PR number/url from the `gh pr create` output). THEN hand back — report the PR URL, files changed, scoped-check result, tests touched, and anything ambiguous. Do NOT run the full gate, do NOT wait for it, do NOT mark your own PR ready — a runner gates it and flips it ready. Never leave committed work unpushed and never leave a pushed branch without a draft PR + an enqueued ticket.

- **No AI attribution.** The harness's default is to add a `Co-Authored-By: Claude …` trailer and a "🤖 Generated with Claude Code" PR line; this workflow overrides that default. Paste this into every brief:

  > No AI attribution: no `Co-Authored-By: Claude` trailer on any commit, no "Generated with Claude Code" line in the PR body. Attribute solely to the configured git user.

### Dispatch in the background, then monitor for divergence
**Default to dispatching implementers in the background** (the Agent tool's `run_in_background: true`) rather than blocking on each one. This keeps you — the orchestrator — responsive: you get an automatic notification when an agent completes or opens its PR, AND you can poll the live worktrees mid-flight to catch a wandering agent *before* it burns a whole run going the wrong way. (Background dispatch is the Agent tool's own flag — this is NOT the banned `isolation: "worktree"`; you still make the worktree yourself via the helper and point a plain agent at it.)

**Poll every ~10 minutes for divergence** while agents run. Use `ScheduleWakeup` to self-pace the checks (≈600s is frequent enough to course-correct early). Don't poll *only* for completion — that arrives as a notification for free; the point of polling is **early divergence detection**, with two cheap riders on the same tick: (a) check whether any slice opened a draft PR and enqueued, and (b) **drain the gate queue** (`DRAIN_CMD`) so enqueued PRs get gated and flipped ready without waiting for you to notice. Each tick, snapshot what files each agent is actually touching and compare against its assigned scope:

- **Snapshot against the FORK POINT (the merge-base), never HEAD and never the integration tip.** Compute the fork point ONCE per tick — `FP=$(git -C <wt> merge-base HEAD origin/<integration>)` (re-`fetch` first; the tip moves) — then `git -C <wt> --no-pager diff --stat $FP` for the full file scope, plus `git -C <wt> ls-files --others --exclude-standard` for untracked. `git diff --stat $FP` (working tree vs fork point) captures BOTH any already-committed blocks AND uncommitted work-in-progress in one shot — which is what you need, because implementers **commit LAST** (after simplify), so mid-flight most of the work is still *uncommitted*; diffing the working tree against the fork point sees it regardless of whether it's been committed yet. Two traps, both of which have actually bitten us, so neither is hypothetical:
  - ⚠️ **Don't use `git diff --stat HEAD`** — that shows only *uncommitted* changes, so the moment an agent commits (it commits everything at the end) HEAD goes clean and the diff shows **nothing**, and you'll either think the agent is idle or reach for a range diff against the tip (the next trap). Use `git log --oneline $FP..HEAD` only to *list the commits*, never as your file-scope check.
  - ⚠️ **Don't use `git diff <integration>..HEAD` / `origin/<integration>..HEAD`** (two-dot against the tip). Once another PR merges into the integration branch *while this worktree is open*, that merged work shows up as **phantom additions/deletions** here and triggers a false divergence alarm — the merged PR's unrelated files look like this agent's edits. We hit exactly this: a poll two-dot-diffed against a tip that had just advanced, the unrelated merged PR's files appeared as the agent's, and a **perfectly in-scope agent was wrongly `TaskStop`ped** (then had to be resumed). The fork point never moves — anchor every scope check to `$FP`, same as the PR-review diff (below).
- **What counts as divergence:** editing another task's / another phase's **core source files** (e.g. a "types only" task editing the resolver, or a backend task building UI). **What does NOT count:** compile-driven ripples from the task's own change — exhaustive `switch`/enum/interface/config entries a new member forces, a one-line badge to keep an exhaustive UI match compiling, and `+1`-line fixture edits across many `*.test.ts` files (even ones in another area) from adding a field to a shared type. Spell out this divergence/not-divergence line in the wakeup prompt so the check is mechanical, not a re-judgement each time.
- **Check substance, not just the file count.** Staying in scope ≠ doing it right — a diffstat tells you *where* the agent is editing, not *whether the approach is correct*. On any slice with real design subtlety, periodically **read the actual diff of the 2–3 highest-risk files** (`git -C <wt> --no-pager diff $FP -- <file>`, anchored to the same fork point as the diffstat — NOT `HEAD`, which would omit the committed blocks), not just `--stat`, and confirm the agent is following the intended approach (or genuinely improving on it). **Grep the diff for the specific anti-patterns the brief banned** (e.g. a forbidden call leaking into the wrong layer: `git -C <wt> diff $FP -- <files> | grep -nE '<bannedSymbols>'`). A wrong-but-in-scope approach is exactly what file-counting misses, and catching it mid-flight saves a whole wasted run. Keep it light — the risk files, not the entire diff.
- **If an agent diverged** (out of scope OR on a wrong approach): `TaskStop` it immediately, then **dispatch a fresh plain agent into the SAME worktree path** with a tighter brief — explicit *do-not-touch* boundaries naming the files/areas it strayed into (or the wrong pattern to avoid), and an instruction to revert the bad edits. Don't try to course-correct a still-running agent mid-stream; stop-and-replace is cleaner.
- **If an agent opened its draft PR + enqueued or completed:** stop rescheduling for that one and move into the drain + PR review loop + merge below.

Carry forward a one-line note of each agent's last-known on-scope file set in the next wakeup prompt, so a sudden jump in surface area is obvious tick-over-tick.

### Draining the gate queue
On each `ScheduleWakeup` tick (the *same* timer you already run for divergence), run `DRAIN_CMD` (Trinity: `pnpm gate:drain`) from the main checkout. One pass claims queued tickets and, for each, runs the full `GATE_CMD` in that ticket's worktree behind the slim machine-wide slot, then flips the PR ready (green) or comments the failure and leaves it draft (red). It is a one-shot pass — it drains what's queued and exits — so the tick re-invokes it.

- **A full drain can be long** (each ticket is one serialized full gate). For a big queue, either bound it (`DRAIN_CMD --max N` per tick) or run the drain with the harness `run_in_background` flag and treat its completion as a resume signal. Don't fire it and then abandon the loop.
- **Multiple orchestrators may drain concurrently** — that's safe by construction (atomic-rename claims, dead-claimant reclaim). You don't coordinate; you both just drain.
- **Don't hand-run `GATE_CMD` on top of a live drain.** The runner already serializes gates one-at-a-time behind the slot; launching your own concurrent `GATE_CMD` re-creates the saturation the slot exists to prevent. Let the runner be the only thing gating.
- **A red ticket is orchestrator feedback, not lost work.** The PR stays a draft with the failure commented. Read it, dispatch a fix agent into that same worktree, and the fix re-pushes and re-enqueues. Nothing is stranded.
- **Reconcile the local integration branch on every tick — the cheap backstop for a dropped sync.** Even though `merge-pr.sh` fast-forwards the local integration branch at merge time, a hand-run merge or an interrupted close-out can leave it behind the remote, and the next worktree would then fork off a stale HEAD. On the same tick as the drain, run one anchored fast-forward against the main checkout: `git -C <main-checkout> fetch origin && git -C <main-checkout> pull --prune --ff-only` (a no-op when already current). It's idempotent and near-instant, so any missed post-merge sync self-heals within one poll instead of poisoning the next dispatch.

### The PR review loop
Each implementer opens **its own draft PR** back to the integration branch; the runner flips it ready when the gate passes. A green (ready) PR is the gate signal — but you still **review each PR and actually read the code involved**, not just the agent's summary or the green gate. Read the diff: verify correctness, that it does what the issue/plan intended, that the scope is right, and scrutinize anything the agent flagged or where it went wider than the planned files. A green gate and a confident agent report are necessary but **not sufficient** — the merge decision is yours and must be grounded in the actual changed code.

**Review the implications, not just the brief.** A PR can match its slice's brief, pass its own gate, and stay perfectly in scope — and still be wrong once it lands among its siblings. Read for *second-order* effects, not only local correctness: (1) **cross-slice composition** — when two slices each own one half of a contract (slice A stamps a header, slice B reads that header for the credential; one slice renames/derives a field, another consumes it), confirm the halves still agree *on the merged branch*, even though the files are disjoint. (2) **shared global state** — a change that's correct in isolation can be silently defeated by a process-wide cache, singleton, or memo it didn't author (does that cache key on the value you just made per-tenant, or globally?). (3) **emergent interactions** — a fix that corrects one path can expose or shift a neighbor's behavior (tightening identity makes a queue correctly *wait* instead of mis-running; a now-required field surfaces a caller nobody listed). File-disjointness buys *textual* independence, never *semantic* independence — so the review asks "what does this imply for everything it touches indirectly?", not just "does it match the brief?". When a second-order effect is load-bearing, go read the shared dependency yourself (the cache, the consumer, the sibling's half) rather than assuming it composes.

**Diff against the merge-base, not the integration tip.** The integration branch advances *while a worktree is open* (other agents' PRs merging in parallel), so `git diff <integration-branch>..HEAD` shows the OTHER work's additions as phantom **deletions** in this PR — a clean change then looks like it ripped out unrelated code, and you'll raise a false alarm. Always diff the fork point: `git -C <wt> diff $(git merge-base HEAD origin/<integration>)..HEAD` (re-`fetch` first — the tip moves). Confirm by test-merging the current base in: if the "deleted" files come back as *additions*, the PR never touched them. This bit us once — burned a review cycle chasing a "destructive deletion" that was just the base having moved.

If it needs changes, **dispatch a fix agent into that same worktree**; it re-pushes and re-enqueues, and the runner re-gates. Iterate until the PR is good and green. Only when satisfied do you **merge and clean up** — and the *order* matters because of worktrees (see next section): the implementer has already pushed everything, so you **remove the worktree first**, then merge with `gh pr merge --merge --delete-branch`.

### Merging a shared hotspot — `mergeable` ≠ correct
When several slices edited the **same function or structure** (a control loop's `tick()`, a reducer, a registry, an exhaustive switch — anything the decomposition flagged as a *shared hotspot*), a clean auto-merge is **not** proof of correctness. Git merges by line: two slices that added code in *different* spots produce `mergeable: MERGEABLE` with no conflict markers — but git has no idea the two additions must **compose**. The recurring failure shape: a structure has sibling branches that must stay **twins** (e.g. two "parked → freeze every kill-clock" branches), each slice added its new clock's freeze-line only to the branches that existed *when it forked*, and the merge silently leaves the newer branch missing it — textually clean, behaviorally a reaper that kills a healthy job. **The integration gate won't catch it** unless a test exercises that exact cross-slice interaction, which is precisely the case nobody wrote.

So whenever a merge — auto-clean OR conflicted — touches a structure 2+ slices edited: **open the merged region and read it; do not trust `mergeable`.** Diff the parallel-edited siblings against *each other* — whatever should be parallel (twin branches, every case of a switch, all handlers of an event, the reducer vs the renderer) must have received the **same** treatment from each slice. Honor the **shared invariant** the decomposition handed you, if any. Fix drift forward as a small, clearly-messaged merge-reconcile commit, then re-verify the interaction (enqueue an integration gate, or run one directly once the fleet is quiet). Resolve actual conflicts by **merging the base into the branch in its worktree** (`git fetch && git merge origin/<integration>`, keep BOTH concerns) — never rebase — then push and `gh pr merge`. This bit us twice in one governor epic: two separate textually-clean auto-merges each dropped a parked-state freeze-line, either of which would have reaped a healthy queued job.

### On the gate — the runner gates; you read the diff
**The RUNNER runs `GATE_CMD` once per PR — that IS the gate.** `GATE_CMD` is the HEAVY full suite: build + the entire test run, serialized behind the slim machine-wide slot (`scripts/gate-slot.mjs` for Trinity) so concurrent gates can't saturate the box. Implementers don't run it (their commits are held only to the cheap scoped check by the pre-commit hook), and release-branch PRs typically get no CI — so the runner's drained gate is what stands in for CI.

**Per-PR, the orchestrator reads the diff — it does NOT hand-re-run the heavy gate.** The runner's green (ready) flip is the per-PR signal; don't duplicate it. Re-gate (by re-enqueuing, or running `GATE_CMD` yourself once the fleet is quiet) only for a genuine reason: the runner never got to a ticket, or a **merge integrates branches that weren't tested together** (one integration gate on the merged result). Even then prefer the minimum — the affected/changed tests over the whole suite where possible. Your always-on job is reading the **diff**; heavy re-gating is conditional.

**A stale slot self-heals; you rarely touch it.** The slot's holder-PID is liveness-checked, so a crashed gate's slot is stolen by the next drain within a poll, and a runner that dies mid-gate has its ticket reclaimed and re-gated. There is no "strangling orphan" for you to clear by hand and no lock dir to `rm` — the queue and slot handle their own dead work. (If you ever do need to forcibly stop a runner's gate, clear its worktree via `~/.worktrees/remove-worktree.sh <leaf>`, which signals the processes; never `rm` the slot dir out from under a possibly-live holder.)

### Transient-red window (schema-first / all-or-nothing epics)
When a foundational change lands first — a NOT-NULL schema swap, a required interface field, a renamed module — it makes `GATE_CMD` impossible to get fully green branch-wide until every consumer migrates. A full-green gate is **unattainable mid-epic BY DESIGN**; holding slices to it just paralyzes progress.

In that window a drained ticket's gate will exit non-zero on the pre-existing baseline failures until every consumer migrates, and the runner will comment the failure and leave the PR draft. That's expected, **not** a real block. Read the runner's output as: the compile half (typecheck + lint) is GREEN, the task's OWN new/affected tests are GREEN, and there are NO failures beyond the baseline SET that pre-dated this task on its fork point (file + test names, not a count — fixing one stale test and breaking a new one keeps the count identical but is still a regression; the absolute count drifts between tasks forked at different tips and as consumers migrate). If it clears that bar, mark the PR ready yourself and merge; don't chase a green exit mid-epic.

**N green PRs ≠ a green epic — gate the integrated whole before you close.** Each branch was gated against *its own fork point*, blind to the other slices; the merged integration branch with all of them together has been gated by *no one*. So a multi-slice epic is NOT done when every PR is individually merged-green — the close-out is: once the last slice merges and the fleet is quiet, run ONE `GATE_CMD` on the integration branch itself (directly, or enqueue a ticket for it) and require it GREEN (modulo a known pre-existing baseline) BEFORE declaring the work done or closing the issue. A failure that first appears here is a real cross-slice interaction the per-PR gates *structurally could not* catch (none of them saw the others) — fix it forward, don't rationalize it away. Independent-green is necessary; *integrated*-green is what actually ships.

### Merge & cleanup — one command, because the sync is the step that gets dropped
The four steps below have exactly one correct order, and step 3 (sync the local integration branch) is the one with **no forcing feedback** — every visible signal after the merge (`✓ Merged`, branch deleted, PR closed) says "done", so it's silently skipped, and the miss only surfaces later when the *next* worktree is cut from a stale HEAD. So **default to the helper that does all four atomically: `~/.worktrees/merge-pr.sh <pr-number>`.** It resolves the main checkout + the PR's base/head branches, tears down the worktree, `gh pr merge --merge --delete-branch`, and — anchored to the main checkout regardless of your cwd — fast-forwards the local integration branch. It's idempotent (a re-run finishes a half-done close-out) and it's the reason a hand-run sync can't get dropped: don't split these steps back out by hand unless the helper can't handle a case.

The branch lives in a linked worktree, and **git refuses to delete a branch that's checked out in a worktree** (`cannot delete branch 'X' used by worktree …`). So `gh pr merge --delete-branch` would error on the local-branch step if the worktree still exists — which is why the order below is load-bearing. If you ever run it by hand (helper unavailable, or an edge it can't cover), do it in this exact sequence, every time:
1. **Confirm the ticket is done before you tear down.** The runner gates in the ticket's worktree, so don't remove a worktree whose ticket is still `queue`/`processing` — wait until its gate has resolved (PR flipped ready, or you've decided to merge). Then **remove the worktree — always via the helper:** `~/.worktrees/remove-worktree.sh <branch-leaf>` (or the absolute path). **Do not use `git worktree remove` directly.** Plain `git worktree remove` evicts the directory but does NOT signal any running processes — so a gate running in it survives as an orphan. The helper kills all processes rooted in the worktree first (via `lsof` + `pgrep`), escalating SIGTERM → SIGKILL, then removes the worktree. Nothing is lost — it's all pushed — and this frees the branch so it can be deleted.
2. **Merge with branch deletion:** `gh pr merge --merge --delete-branch` — real merge commit (never squash), and with the worktree gone `--delete-branch` cleanly removes BOTH the local and remote branch. Skipping `--delete-branch` is what let merged `origin/*` branches pile up indefinitely (they did: dozens accumulated). Verify it's fully merged first; STOP on git's "not yet merged to HEAD" warning.
3. **Sync the local integration branch — anchored to the MAIN checkout, never `&&`-chained.** `gh pr merge` advances the branch only on the *remote*; the local integration branch in the main checkout does NOT move, and syncing it is what keeps the next worktree from forking off a stale HEAD. Run it anchored so cwd can't defeat it: `git -C <main-checkout> checkout <integration>` then, as a **separate** command, `git -C <main-checkout> pull --prune --ff-only`. ⚠️ Do NOT hand-run the old `git checkout <integration> && git pull` form: with no `-C` anchor it assumes your shell is in the main checkout, but the orchestrator's cwd is usually inside a worktree — there `git checkout <integration>` fails with `already used by worktree at …`, and the `&&` swallows the `git pull` so the sync silently never happens (this is exactly the "merged but never synced" bug). The two-command, `-C`-anchored form is why the helper above can't drop it.
4. **Close the issue yourself if the work resolved one — GitHub will NOT auto-close it.** Auto-close only fires when a PR with a `Closes #N` keyword merges into the **default branch** (`main`). Our PRs merge into the **integration branch**, so a linked issue stays open regardless of keywords — and in practice the PRs only cross-reference (`#N`) rather than use a closing keyword anyway, so no close link even exists. After merging, `gh issue close <N> --comment "Fixed in #<pr> (merged into <integration>)."` for each issue the merged work resolved. (Don't wait for the eventual integration→`main` merge to sweep them — close them now so the issue tracker reflects reality.)

## Implementer

Work only in the assigned worktree (absolute paths), match surrounding style, and follow the project's conventions — its `BRIEF_CONVENTIONS` (from `~/.worktrees/config/<project>.sh`) and its `AGENTS.md`. (For Trinity — **app = Solid, sidecar = Effect**: invoke the `effect` skill for any Effect-TS code on the sidecar, and the `solid` skill for any SolidJS UI code on the app — `solid-js`, `@solidjs/router`, `@tanstack/solid-query`, Kobalte; build forward-only — pre-launch, no backwards-compat shims; comments explain the mechanism, no issue/PR/version/plan refs.)

**The flow is: write all the code → (`/simplify` if this slice warrants it) → commit → push → draft PR → enqueue the gate → hand back.** Write the whole change in your worktree (running *cheap* checks as you work is good — format, a scoped lint/typecheck of the file or package you're in, a single targeted test file run directly). For anything wider than one file — a whole package or the whole suite — go through the **cached runner** (`SCOPED_CHECK_CMD`, or `turbo run <task> --filter=<pkg>`), never a raw `vitest`/`tsc` or a per-package script that shells straight to the binary; those bypass the shared cache and run cold every time.

**Whether you run `/simplify` is the orchestrator's call, made per slice in your brief** — it's a quality pass worth its round-trip on substantial work and pure noise on trivial changes. If the brief says to simplify (the default for substantial slices), do NOT commit as you go: keep the whole change uncommitted, and when the code is done run the `/simplify` skill over your *full uncommitted* working-tree diff (reuse / simplification / efficiency / altitude cleanups — quality only, not bug-hunting), fold the fixes in, and ONLY then commit. If the brief says to skip it — or you're a direct implementer with no orchestrator and the change is plainly trivial (a one-liner, a copy/config tweak, a mechanical rename) — skip `/simplify` and commit in logical blocks as the work lands; don't run it just to report "nothing actionable."

⚠️ **Why commit last when you DO simplify:** `/simplify` acts on the *uncommitted* working-tree diff. Anything you've already committed is invisible to it — so if you commit as you go, the earlier blocks never get simplified, and against a fully-committed (clean) tree `/simplify` sees an empty diff, finds nothing, and silently does nothing while reporting "nothing actionable." The review simply never happens. Keeping everything uncommitted until after simplify is what makes the pass actually cover the whole change. (A skip-simplify slice has no such constraint — commit as the blocks land.)

**Always format in write mode right before committing.** The scoped check runs format-*check* (Trinity: `pnpm format` in write mode vs. the check inside `pnpm check`) — the pre-commit hook verifies formatting, it does not fix it. So run the auto-formatter over your tree as the last thing before `git commit`; then the hook's format-check passes on the first try instead of bouncing you on whitespace you never saw. Cheap, and it removes a whole class of surprise commit denials.

**Formatter output is always committed, never reverted.** When the auto-formatter (or the gate, which runs it) rewrites a file, commit the result — do not revert it as "out of scope." Formatting of the files *you* changed folds into your change; formatting of files *unrelated* to your change (a pre-existing non-canonical file the formatter normalized) goes in its OWN `chore(format)` commit. And when you run the full gate yourself (override mode), it formats *in your worktree* — so on green you CAN and SHOULD commit whatever it wrote and push, rather than re-running the slow gate to re-verify formatting (pure prettier writes can't break a check/test that already passed). The one thing you must never do is throw formatter output away: a reverted format is drift left on the branch for the next agent to trip over.

**You do NOT run the full gate.** The heavy build+test gate runs later, in a runner, drained from the queue — not by you. Your commits are held only to a cheap **scoped check** (`SCOPED_CHECK_CMD` = format-check + lint + typecheck; no build, no test suite), and the agentic pre-commit hook already enforces it on every `git commit` — so a commit that would fail the scoped check is denied and you fix it and re-commit. That's the entire quality bar you personally clear. Never run `GATE_CMD` yourself, never wait on a gate, never mark your own PR ready. (Exception: if the orchestrator's brief put this slice in **override gate mode** — foundational/cross-cutting, or no orchestrator is draining — then run the full `GATE_CMD` yourself once, flip your own PR ready on green, and do NOT enqueue. That is the only path where you touch the full gate.)

**Then commit in logical blocks** — committing last loses nothing: stage each self-contained step (a type + its plumbing, a behavior + its tests, one cohesive refactor) into its own commit with a mechanism-explaining message (`git add -p` / path-scoped staging), so the never-squashed history stays reviewable block-by-block — now over code that's already simplified. **Never rebase. Never merge your own PR.**

**⛔ Attribute every commit + PR to the git user ONLY — no Claude attribution, ever.** This OVERRIDES the harness default that tells you to end commit messages with a `Co-Authored-By: Claude …` trailer — that default does NOT apply here. Do NOT add a `Co-Authored-By: Claude …` trailer (any model) to commit messages, and do NOT add a "🤖 Generated with Claude Code" line to PR bodies. Keep messages clean and attributed solely to the configured git user.

**The handoff — push, draft PR, enqueue, hand back.** This exact order is what makes a mid-flight death lose nothing: the branch is pushed and the PR is open before the ticket exists, so if you die right after enqueuing, your work is a visible draft PR that a runner will still gate.
1. **Push** all your commits.
2. **Open a DRAFT PR** targeting the integration branch (`gh pr create --draft`). Capture the PR number and URL from the output.
3. **Enqueue the gate:** `ENQUEUE_CMD --branch <yourBranch> --worktree <yourWorktreeAbsPath> --pr-number <n> --pr-url <url>` (Trinity: `pnpm gate:enqueue …`). This drops a durable on-disk ticket; a runner will drain it, run the full gate in your worktree, and flip the PR ready (green) or comment the failure (red).
4. **Hand back** — report the PR URL, files changed, scoped-check result, tests touched, and anything ambiguous. You're done; you do NOT wait for the gate.

**Don't be lazy — leave the tree cleaner than you found it, and commit out-of-scope fixes separately.** Your job is not "make the assigned change and stop the instant the scoped check stops *erroring*." A check that exits green but still prints warnings is NOT done. Actively fix every lint warning, type warning, and error you hit or that your own change surfaces — even when it is technically out of scope: the residual unused imports a file is left with after you move its code to a sibling, a path-keyed lint allowlist (sanctioned-git / ambient-identity / ambient-network rules) that must follow the relocated code to its NEW file path, a stray nit in a file you only opened to read. The bar is **zero warnings and zero errors on every file your change touches.** **Isolate each such opportunistic / out-of-scope fix in its OWN commit** with a clear mechanism-explaining message (e.g. a `chore(lint): prune imports the split left unused` commit, kept separate from the `refactor: split …` commits) — so it stays independently reviewable and the orchestrator can keep or drop it without untangling it from the task. Never silently fold an unrelated fix into a task commit, and never hand back a "green" PR that still carries warnings for someone else to clean up. (If a surfaced fix is genuinely large, risky, or a behavior change, still isolate it in its own commit AND call it out in your report — don't bury it, and don't skip it.)

**If you get stuck, hand back a reviewable artifact — never spin or die silently.** A blocker, an ambiguity you can't resolve, a scoped check you can't get green, or running low on room all mean the same thing: STOP and give the orchestrator something it can act on. Commit-push and open a **draft PR** with what you have (and enqueue it if it's far enough along to gate); if you can't even do that, hand back a complete report of your worktree state — what's done, what's broken, the exact error/blocker, and the decision you need. Partial-but-clean beats stuck-and-silent (e.g. land the solid part and flag the rest). **Never leave a dirty worktree with no PR and no report, and never leave a committed-but-unpushed branch — push and open a DRAFT PR so the orchestrator inherits a reviewable, mergeable artifact rather than invisible work trapped in your worktree.**

## Hard rules (both roles)

- **⛔ Implementers never run the full gate; they enqueue it.** The heavy build+test `GATE_CMD` runs one-at-a-time in a runner drained from the durable queue, behind a slim machine-wide slot. An implementer's quality bar is the cheap scoped check (enforced by the pre-commit hook); its wrap-up is push → draft PR → enqueue. The only exception is an orchestrator-flagged **override** slice (or no orchestrator to drain), where the implementer runs `GATE_CMD` itself and flips its own PR ready. Never hand-run a `GATE_CMD` concurrently with a live drain — the runner already serializes gates; a concurrent one re-creates the saturation the slot prevents.
- **⛔ Never use the Agent tool's `isolation: "worktree"` param or any auto worktree provisioner.** Make worktrees only via `~/.worktrees/setup-worktree.sh`, verify HEAD, then dispatch a plain agent. (See worktree section.)
- **Never squash-merge; merge + clean up + sync as one step.** Prefer `~/.worktrees/merge-pr.sh <pr-number>`, which does the whole close-out atomically (tear down worktree → real merge commit → fast-forward the local integration branch). If you merge by hand, use a real merge commit (`gh pr merge --merge --delete-branch`), never `--squash` — squashing flattens the individual commits we keep. Always pass `--delete-branch` so the remote branch is removed on merge instead of accumulating — but **remove the worktree first** (see Merge & cleanup), or `--delete-branch` errors trying to delete a branch still checked out in a worktree. **And always finish with the local integration-branch sync**, anchored to the main checkout (`git -C <main-checkout> pull --prune --ff-only`) — never the unanchored `git checkout <integration> && git pull`, whose `&&` silently swallows the pull when your cwd is inside a worktree, leaving the branch merged-on-remote but stale locally and the next worktree forked off a stale HEAD.
- **Do NOT rebase.** Rebasing rewrites history. Default to plain merge commits everywhere: integrating parallel worktree branches, a branch falling behind the integration branch, resolving overlap between two PRs — MERGE, never rebase. If two parallel branches touch the same file, resolve it at MERGE time (merge one, then merge the other and fix conflicts in the merge), not by rebasing one onto the other. Never instruct a sub-agent to rebase. Only rebase when there's a specific, unavoidable reason AND it's worth rewriting history — rare; when in doubt, merge.
- **⛔ Whether to run `/simplify` is the orchestrator's per-slice call — and WHEN it runs, commit LAST.** Simplify is a quality pass worth its round-trip on substantial work and noise on trivial changes (one-liners, copy/config tweaks, mechanical renames), so the orchestrator decides per slice and states it in the brief (a direct implementer with no orchestrator makes the same call for itself). When a slice simplifies, the order is load-bearing: `/simplify` reads the working-tree diff, so against a clean (fully-committed) tree it silently no-ops — finding "nothing actionable" because there's nothing to see, not because the code is clean. So write all the code → `/simplify` → *then* commit (in logical blocks) → push → draft PR → enqueue; never commit as you go and never commit-everything-then-simplify, either of which hides code from the pass. A skip-simplify slice just commits in logical blocks as it goes. Cheap checks (format, scoped lint/typecheck, a single test file) may run mid-task regardless.
- **⛔ No Claude attribution on commits or PRs — git user only.** This OVERRIDES the harness's built-in "end commit messages with `Co-Authored-By: Claude …`" instruction; it does not apply in this flow. Never add a `Co-Authored-By: Claude …` trailer (any model) to a commit, nor a "🤖 Generated with Claude Code" line to a PR body. Attribute solely to the configured git user. Spell this out in every implementer brief — dispatched sub-agents follow the harness default unless told otherwise.
- **Branch from the active integration branch, not `main`.** PRs target the integration branch.
- **Never delete a branch past git's "not merged" warning** — verify fully merged into the target first.
- **⛔ Don't bypass the shared build cache — run tasks through the runner.** Cache-eligible tasks (test, typecheck, lint) must go through the task-runner — the project's `SCOPED_CHECK_CMD`/`GATE_CMD`, or `turbo run <task> --filter=<pkg>` — NOT the raw binary (`vitest`, `tsc`, `eslint`) and NOT a per-package script that shells straight to it (`pnpm --filter <pkg> test` when that script is a bare `vitest run`). A direct run always goes cold — locally *and* in the drained gate — and never populates the cache for the next machine/CI. The only sanctioned direct-binary run is a **single targeted test file**. If a project's `AGENTS.md` documents the raw form as the whole-package default, that's a leak to fix, not a pattern to copy into briefs.

**Why this shape:** integration branch as the merge point, isolated worktrees per task, clean up after merge. The heavy gate is decoupled from the flaky sub-agent that produced the change — implementers enqueue a durable ticket and hand back; robust orchestrators drain the queue and gate one-at-a-time — so a dying agent never strands committed work and a wide fan-out never serializes on a lock. We preserve real merge history rather than flattening it via squash or rebase. Skipping the post-merge local sync leaves the local integration branch behind the remote, so the next worktree gets cut from a stale HEAD.
